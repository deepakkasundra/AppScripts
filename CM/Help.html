Setting up Configuration

First steps is to set configuration to start CM verification

  1. To start CM verification, follow these instructions:
  2. Ensure you have selected the Region in the "Main" sheet.
  3. Enter Dashboard Credentials in the Main sheet.
  4. Set NLP User Name and NLP User Password for NLP triaging verification.

Validate Ticketing Forms

This will ensure that the combination of category and sub-category is mentioned correctly in the Ticketing form.

Perform the following steps to check Ticketing forms:

  1. Generate a token to download data from the Dashboard.
  2. Go to Fetch Dashboard data > and click on Generate Token for the respective environment.
  3. Once the token is successfully generated, go to QA Verification Menu > Check Ticketing Form > Fetch Ticketing form i.e. UAT (New CM) or Fetch Ticketing Form from PROD (New CM).
  4. You will get an S3 downloadable link in the "TicketingForm" sheet. (The download link will expire in 30 mins or as per configured time.)
  5. Validate the ticketing form from Menu > QA Verification > Check Ticketing Form > Import All Forms and Validate All Forms.
  6. Once you click the above option, it will fetch Ticketing forms data and create two sheets. One is for the original form and the second with Validation.
  7. Validation statuses will appear in the "Ticketing form verification" sheet, and consolidated data will appear in "Exported Data from ticketing forms".
  8. If you find the status as "Missing in Column options; It's critical," it means the Category is not mapped properly. In this case, connect with BA ASAP and raise the issue.
  9. If you find the status as "Missing in Column dependsOnValue," it means there is no Sub Category available for the mentioned Category.

Separate Exported Forms

Extract and Convert Ticketing Form Data

  1. This activity will separate category and subcategory in a comparison format.
  2. If you opted for "Import All Forms and Validate All Forms," no manual work is needed.
  3. To convert data, go to QA Verification Menu > Separate Exported Data > select the available option.
  4. Once the script completes successfully, all categories and sub-categories will be available in the "Separated Data" sheet.
  5. Based on Extra fields, we have the Assignee matrix.

    6. What is the use of Extra Field here?

    • In CM, we can configure different Assignees based on extra parameters such as Grade, Institute, Level, Relationship, etc.
    • If any extra parameter is used in the Ticketing form (e.g., extraFields.{FIELDNAME}), this script helps to convert the data for comparison.
    • Use Case - In the ticketing form, we have configured extraFields.Institute. Based on the Institute, there are Assignees. For example, for Institute = ABC, it will assign that ticket to Assignee A; if Institute = XYZ, it will assign to Assignee B.

Ticketing Form vs Assignee Config

Comparison of Ticketing Form vs. Assignee Config

  1. This comparison ensures that all categories available in the Ticketing form are present in the Assignee Config.
  2. To start verification, click on QA Verification Menu > Ticketing form (Separated Data) vs Assignee Config > Select the respective available options.
  3. If you have extra fields based on Assignee config, select "Ticketing forms vs. {Respective sheet} with Param" or run "Ticketing forms vs. {Respective sheet} without Param."
  4. Once the execution is complete, the report will be generated in the "Ticketing_Form_vs_{sheet_Name}_Assignee Config" sheet. Refer to the "Combination Status" column for the status.

    5. Note:

  5. There are two statuses: "Not found in Assignee config" and "Not found in Ticketing Form."
  6. If you receive the status "Not found in Assignee config," it means the category combination is available in the ticketing form but not present in the Assignee config. This is critical and needs to be fixed ASAP; otherwise, the ticket will not be assigned to any agents.
  7. If you receive the status "Not found in Ticketing Form," it means the category combination is available in the Assignee config but not in the Ticketing form. This may be the case when extra category combinations are added or forgotten to remove from the Assignee config.

Assignee Config Comparison

Here we compare Requirement sheet with UAT assignee config.

  1. This comparisonensure that the configuration of assignement is as per requirement sheet that is provided by client.
  2. To start verification, click on QA Verification Menu > Assignee Config Comparison > Select the respective available options.
  3. Here we can compare complex assignee config, if in assignee config based on any extra param like Institution code, Employee Location, Employee Country etc. you can opt. out Compare Requirment vs. UAT SLA with Param.

    4. Note :- When you select Compare Requirment vs. UAT SLA with Param you need to maintain param in respective sheet in column "Execution Rule 1" with same sequence.

    If you have multiple param like Country, Location, Grade etc. you just simply add all the extra param in that column by below formula.
    Column1 = Location
    Column2 = Country
    Column3 = Grade
    Formula =TRIM(Column1&" "&Column2" "&Column3)



For Example you have below Param:


Execution Rule 1 Location Country Grade
=TRIM(Column1 & " " & Column2 & " " & Column3) Mumbai India L1




It will display the value in Execution Rule 1 as below:

Execution Rule 1 Location Country Grade
Mumbai India L1 Mumbai India L1

Separate Sub Categories by Semi-Colon

If the Sub Categories in the Assignee Config are separated by a semi-colon, you can use this option to separate them:

  1. Go to the ⚙️ QA Verification > select Separate Sub Categories by Semi-Colon > select the respective option.

Refer to the example below: Where multiple Sub Categories are mentioned in a single cell
